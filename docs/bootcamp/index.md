### 1. 스프링, 스프링 부트와의 첫 만남

- 프레임워크와 라이브러리는 모두 다른 사람이 작성한 코드다.
- 프레임워크와 라이브러리의 가장 큰 차이점은 코드를 호출하는 지점(제어 흐름)이다.
- 프레임워크는 정해진 흐름 속에서 사용자의 코드를 호출한다. 라이브러리는 사용자가 직접 호출한다.
- 스프링부트(Spring Boot)는 스프링(Spring)을 설정 중심에서 사용 중심으로 변화한 도구다. 복잡한 설정을 자동화로 처리했다.
- 컴퓨터로 비유하면 스프링은 내부 구조, 동작 원리에 가깝고, 스프링 부트는 컴퓨터를 실제 사용하는 경험에 가깝다.
- 컴퓨터를 처음 사용했을 때 CPU, RAM, 회로부터 배웠다면 금방 흥미를 잃었을 것이다. 마찬가지로 스프링도 우선 스프링부트를 통해 사용 경험을 쌓고 자연스럽게 내부 원리(스프링)를 접하는 방법을 권장한다.

### 2. 스프링 부트와 웹

- 인터넷과 웹은 비슷하게 사용한다. 하지만 명백하게 다른 의미다.
- 인터넷은 전 세계 컴퓨터를 연결하는 거대한 네트워크 인프라다.
- 웹은 인터넷을 사용하여 정보를 공유한다.
- 웹은 인터넷 위에서 동작하는 수많은 서비스 중 하나다.
- 인터넷이 도로라면 웹은 인터넷 위에서 달리는 자동차와 같다.
- HTTP는 HTML 문서와 같은 리소스를 가져올 때 사용하는 프로토콜이다.

### 3. 보이지 않는 곳의 핵심 기능, API

- API란 컴퓨터와 컴퓨터를 연결하는 인터페이스다.
- API는 내부 구현, 세부 사항을 숨기고 필요한 부분만 제공하여 변경되더라도 일관성을 유지한다.
- REST(Representational State Transfer)는 소프트웨어 아키텍처 형식이다.
- 자원을 이름으로 구분하여 자원의 상태(정보)를 주고받는다.
- REST는 신뢰할 수 있는 웹 애플리케이션을 만들기 위해 사용한다.
- RESTful이란 REST 아키텍처의 설계 원칙을 준수하여 웹의 장점을 최대한 활용하는 API를 의미한다.

### 4. 스프링 부트 프로젝트의 시작

- 메타데이터 구조
    - Group: 기본 패키지 구조
    - Artifact: 메인 패키지
    - Name: 프로젝트
    - Description: 프로젝트 소개
    - Package name: Group + Artifact

### 10. 개발자라면 응당 테스트

- 응당 테스트란 '개발자라면 응당하게(당연하게) 테스트를 해야 한다는 의미'다. 응답 테스트의 오타가 아니다.
- 테스트
    - 기능의 정상 동작
    - 예외 상황에서의 올바른 응답
    - HTTP 상태 코드
    - 입력 검증
- 브라우저는 `GET` 요청만 쉽게 보낼 수 있다. JSON Body 전송이 불편하다. API 도구를 사용하면 쉽게 테스트를 검증할 수 있다.
- Spring은 보통 수동 API 테스트(Postman) -> 단위 테스트(JUnit + Mockito) -> 통합 테스트 단계를 통해 동작을 검증한다.
- Postman 외에도 HTTPie, IntelliJ HTTP Client를 사용할 수 있다.
